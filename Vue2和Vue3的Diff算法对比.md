# Vue2 和 Vue3 的 Diff 算法对比

## 先搞懂: 什么是 Diff 算法？为什么需要它？

一句话：**Diff 算法就是"找不同"**。

当数据变化时，会生成一棵新的虚拟 DOM 树。Diff 算法的任务就是：**用最少的操作，把旧树变成新树**。

```plaintext
数据变化前的虚拟DOM(旧树)          数据变化后的虚拟DOM(新树)
       div                              div
      / | \           Diff算法          / | \
     A  B  C      ──────────→         A  D  C
                   找出差异: B→D
                   最终只更新B这个节点为D
```

如果不用 Diff，每次数据变化就得把整个 DOM 推倒重建，性能很差。

---

## Vue2 的 Diff 算法: 双端比较

Vue2 使用的是**双端比较(Double-Ended Diff)**算法。

### 核心思路

同时从新旧子节点列表的**两头往中间靠拢**，设置 4 个指针：

```plaintext
旧子节点:  [A]  B  C  [D]
            ↑oldStart    ↑oldEnd

新子节点:  [D]  A  B  [C]
            ↑newStart    ↑newEnd
```

### 比较顺序(每一轮都按这个顺序尝试)

```plaintext
第1步: oldStart vs newStart  (旧头 vs 新头)
第2步: oldEnd   vs newEnd    (旧尾 vs 新尾)
第3步: oldStart vs newEnd    (旧头 vs 新尾)
第4步: oldEnd   vs newStart  (旧尾 vs 新头)
第5步: 以上都不匹配 → 拿newStart的key去旧列表中查找
```

### 走一遍完整流程(用例: 旧 A B C D → 新 D B A C)

这个例子特意设计成 **4轮分别命中4种不同的比较规则**：

```plaintext
目标: 把真实DOM从 [A B C D] 变成 [D B A C]

══════════════════ 初始状态 ══════════════════

旧列表:  A    B    C    D
         ↑oldS              ↑oldE

新列表:  D    B    A    C
         ↑newS              ↑newE

真实DOM: [A] [B] [C] [D]

══════════════════ 第1轮: 命中"旧尾 = 新头" ══════════════════

比较1: oldS(A) vs newS(D) → 不同 ✘
比较2: oldE(D) vs newE(C) → 不同 ✘
比较3: oldS(A) vs newE(C) → 不同 ✘
比较4: oldE(D) vs newS(D) → 相同 ✅ 命中!

规则: "旧尾 = 新头" → 说明D要从最后面跑到最前面
操作: 把D的DOM节点移动到oldS(A)节点的前面
指针: oldE左移一位, newS右移一位

旧列表:  A    B    C   |D已处理|
         ↑oldS    ↑oldE

新列表: |D已处理|  B    A    C
                  ↑newS      ↑newE

真实DOM: [D] [A] [B] [C]    ← D移动到了A前面
          ↑移到这里了

══════════════════ 第2轮: 命中"旧尾 = 新尾" ══════════════════

比较1: oldS(A) vs newS(B) → 不同 ✘
比较2: oldE(C) vs newE(C) → 相同 ✅ 命中!

规则: "旧尾 = 新尾" → C本来就在末尾,新列表末尾也是C,位置对了
操作: 只做patch,不移动DOM
指针: oldE左移一位, newE左移一位

旧列表:  A    B   |C已处理||D已处理|
         ↑oldS ↑oldE

新列表: |D已处理|  B    A  |C已处理|
                  ↑newS ↑newE

真实DOM: [D] [A] [B] [C]    ← 没有变化

══════════════════ 第3轮: 命中"旧头 = 新尾" ══════════════════

比较1: oldS(A) vs newS(B) → 不同 ✘
比较2: oldE(B) vs newE(A) → 不同 ✘
比较3: oldS(A) vs newE(A) → 相同 ✅ 命中!

规则: "旧头 = 新尾" → 说明A要从前面跑到后面去
操作: 把A的DOM节点移动到oldE(B)节点的后面
指针: oldS右移一位, newE左移一位

旧列表: |A已处理| B   |C已处理||D已处理|
                 ↑oldS
                 ↑oldE

新列表: |D已处理|  B  |A已处理||C已处理|
                  ↑newS
                  ↑newE

真实DOM: [D] [B] [A] [C]    ← A移到了B的后面
                 ↑移到这里了

══════════════════ 第4轮: 命中"旧头 = 新头" ══════════════════

比较1: oldS(B) vs newS(B) → 相同 ✅ 命中!

规则: "旧头 = 新头" → B的位置是对的,不需要移动
操作: 只做patch,不移动DOM
指针: oldS右移, oldS > oldE → 结束!

真实DOM: [D] [B] [A] [C]    ← 最终结果,和新列表一致 ✅

总结: 4轮分别命中了4种不同的比较规则,只做了2次DOM移动(D和A)
```

### 再看一个有"新增"和"删除"的例子(旧 A B C D → 新 B E A)

```plaintext
目标: 把真实DOM从 [A B C D] 变成 [B E A]

══════════════════ 初始状态 ══════════════════

旧列表:  A    B    C    D
         ↑oldS              ↑oldE

新列表:  B    E    A
         ↑newS    ↑newE

真实DOM: [A] [B] [C] [D]

══════════════════ 第1轮 ══════════════════

比较1: oldS(A) vs newS(B) → 不同 ✘
比较2: oldE(D) vs newE(A) → 不同 ✘
比较3: oldS(A) vs newE(A) → 相同 ✅ 命中!

规则: "旧头 = 新尾" → 说明A要从前面跑到后面去
操作: 把A的DOM节点移动到oldE(D)节点的后面
指针: oldS右移一位, newE左移一位

旧列表: |A已处理| B    C    D
                 ↑oldS      ↑oldE

新列表:  B    E   |A已处理|
         ↑newS
         ↑newE

真实DOM: [B] [C] [D] [A]    ← A移动到了D的后面
                      ↑移到这里了

══════════════════ 第2轮 ══════════════════

比较1: oldS(B) vs newS(B) → 相同 ✅ 命中!

操作: 只做patch,不移动DOM
指针: oldS右移一位, newS右移一位

旧列表: |A已处理||B已处理| C    D
                         ↑oldS ↑oldE

新列表:  |B已处理| E   |A已处理|
                  ↑newS
                  ↑newE

真实DOM: [B] [C] [D] [A]    ← 没有变化

══════════════════ 第3轮 ══════════════════

比较1: oldS(C) vs newS(E) → 不同 ✘
比较2: oldE(D) vs newE(E) → 不同 ✘
比较3: oldS(C) vs newE(E) → 不同 ✘
比较4: oldE(D) vs newS(E) → 不同 ✘

全部不匹配! 进入第5步:
拿newS(E)的key去旧列表中查找 → 找不到!
说明E是全新的节点 → 创建E的DOM,插入到oldS(C)节点前面
指针: newS右移一位

旧列表: |A已处理||B已处理| C    D
                         ↑oldS ↑oldE

新列表:  |B已处理||E已处理||A已处理|
                  newS > newE → 新列表处理完了!

真实DOM: [B] [E] [C] [D] [A]    ← E插入到了C前面
              ↑新建的

══════════════════ 收尾 ══════════════════

新列表已遍历完(newS > newE),但旧列表还剩 C和D(oldS到oldE)
规则: 旧列表中剩余的节点在新列表中不存在 → 全部删除!

删除C ✘  删除D ✘

真实DOM: [B] [E] [A]    ← 最终结果,和新列表一致 ✅

总结本轮操作: 1次移动(A) + 1次新建(E) + 2次删除(C,D)
```

### Vue2 Diff 的特点

- **同层比较**: 只比较同一层级的节点，不跨层
- **双端指针**: 4个指针从两端向中间收拢
- **key的作用**: 用 key 来判断节点是否是"同一个"，所以 v-for 要写 key
- **时间复杂度**: O(n)

## Vue3 的 Diff 算法: 最长递增子序列

Vue3 在双端比较的基础上做了优化，引入了**最长递增子序列(LIS)**。

### 核心思路: 分三步走

```plaintext
第1步: 从头部开始，跳过相同的前缀
第2步: 从尾部开始，跳过相同的后缀
第3步: 处理中间乱序部分 → 用最长递增子序列(LIS)优化移动
```

### 先搞懂: 什么是最长递增子序列(LIS)?

```plaintext
大白话: 从一串数字中,找出最长的一段"越来越大"的子序列(可以不连续)

举例: 数组 [3, 1, 2, 5, 4]
      递增子序列有: [1,2,5], [1,2,4], [3,5], [1,5] ...
      最长的是: [1, 2, 5] 或 [1, 2, 4]，长度为3

它在Diff里的作用:
  旧列表里有些节点在新列表中相对顺序没变 → 这些不用动
  LIS就是找出"最多有哪些节点不用动"
  剩下的才需要移动 → 移动次数最少!
```

### 完整流程走一遍(用例: 旧 A B C D E F G → 新 A B D E C H G)

```plaintext
目标: 把真实DOM从 [A B C D E F G] 变成 [A B D E C H G]

══════════════════ 第1步: 头部预处理 ══════════════════

从头开始逐个比较,跳过相同的:

旧:  A    B    C    D    E    F    G
     ↑i
新:  A    B    D    E    C    H    G
     ↑i

A === A ✅ → i++
B === B ✅ → i++
C !== D ✘ → 停! i停在2

结果: 头部的A、B不需要处理,直接跳过

══════════════════ 第2步: 尾部预处理 ══════════════════

从尾部开始逐个比较,跳过相同的:

旧:  A    B    C    D    E    F    G
                                   ↑e1
新:  A    B    D    E    C    H    G
                                   ↑e2

G === G ✅ → e1--, e2--
F !== H ✘ → 停! e1停在5(F), e2停在5(H)

结果: 尾部的G不需要处理,直接跳过

══════════════════ 预处理后的状态 ══════════════════

已跳过: A B [......] G
                ↑只需要处理这部分

旧的剩余(索引2~5): C  D  E  F
新的剩余(索引2~5): D  E  C  H

真实DOM: [A] [B] [C] [D] [E] [F] [G]
                  └── 只动这个区间 ──┘

══════════════════ 第3步: 处理中间乱序部分 ══════════════════

─── 3.1 为新节点建立 key→index 映射 ───

新的剩余: D(索引2) E(索引3) C(索引4) H(索引5)

keyToNewIndex = { D:2, E:3, C:4, H:5 }

─── 3.2 遍历旧节点,建立 newIndex→oldIndex 映射数组 ───

创建数组 newIndexToOldIndex,长度=4(新剩余节点个数),初始值全为0
(0表示"这个新节点在旧列表中不存在")

                    对应新节点:  D    E    C    H
newIndexToOldIndex:           [ 0,   0,   0,   0 ]

遍历旧的剩余节点:

  旧C(旧索引2) → 查Map → C在新列表索引4 → 数组[4-2]=数组[2] = 2+1 = 3
  旧D(旧索引3) → 查Map → D在新列表索引2 → 数组[2-2]=数组[0] = 3+1 = 4
  旧E(旧索引4) → 查Map → E在新列表索引3 → 数组[3-2]=数组[1] = 4+1 = 5
  旧F(旧索引5) → 查Map → F不在新列表中  → 删除F! ✘

  (注: 存的是 oldIndex+1,因为0被保留表示"新增")

                    对应新节点:  D    E    C    H
newIndexToOldIndex:           [ 4,   5,   3,   0 ]
                                ↑    ↑    ↑    ↑
                               旧D  旧E  旧C  新的

此时已执行操作: 删除F的DOM节点
真实DOM: [A] [B] [C] [D] [E] [G]    ← F被删了

─── 3.3 求最长递增子序列(LIS) ───

newIndexToOldIndex = [4, 5, 3, 0]

忽略0(代表新增的H),对 [4, 5, 3] 求LIS:
  递增子序列: [4,5] ← 最长! 长度2
  对应位置索引: [0, 1]

意思是: 数组第0位(D)和第1位(E)的相对顺序已经正确,不需要移动!

─── 3.4 从右往左遍历新节点,执行DOM操作 ───

LIS索引 = [0, 1],用一个指针j从LIS末尾开始: j=1

从右往左处理新的剩余节点 D E C H:

  ▸ i=3, 新节点H: newIndexToOldIndex[3]=0 → 是新节点
    操作: 创建H,插入到下一个节点(G)的前面
    真实DOM: [A] [B] [C] [D] [E] [H] [G]
                                  ↑新建

  ▸ i=2, 新节点C: i=2 不在LIS[0,1]中 → 需要移动!
    操作: 把C移动到下一个已处理节点(H)的前面
    真实DOM: [A] [B] [D] [E] [C] [H] [G]
                              ↑C移到这里

  ▸ i=1, 新节点E: i=1 在LIS[0,1]中 → 不移动 ✅
    j--

  ▸ i=0, 新节点D: i=0 在LIS[0,1]中 → 不移动 ✅
    j--

══════════════════ 最终结果 ══════════════════

真实DOM: [A] [B] [D] [E] [C] [H] [G]  ✅ 和新列表一致!

总结本轮操作:
  - 跳过了3个节点(A, B, G)          ← 头尾预处理的功劳
  - D, E不移动                      ← LIS的功劳
  - C移动1次(移到H前面)
  - F删除1次
  - H新建1次
  总共只有3次DOM操作!

如果不用LIS,C、D、E可能都要移动 → LIS省了2次移动
```

## Vue2 vs Vue3 Diff 关键区别对比

| 对比项          | Vue2                  | Vue3                           |
| --------------- | --------------------- | ------------------------------ |
| **算法名称**    | 双端比较              | 预处理 + 最长递增子序列        |
| **比较方式**    | 4个指针两端往中间靠拢 | 先处理首尾相同，再处理中间乱序 |
| **乱序处理**    | 暴力查找/遍历         | 用LIS找出最少移动方案          |
| **DOM移动次数** | 较多                  | 更少(LIS中的节点不用动)        |
| **性能**        | O(n)                  | O(n)，但常数项更小，实际更快   |
| **静态标记**    | 无                    | 有PatchFlag，跳过静态节点      |

## Vue3 额外的编译优化(和Diff配合)

Vue3 不仅优化了 Diff 算法本身，还在**编译阶段**做了标记，让 Diff 可以跳过不必要的比较：

```javascript
// Vue3 编译后的虚拟DOM会带上PatchFlag
{
  type: 'div',
  children: 'hello',
  patchFlag: 1  // 1 = TEXT，表示只有文本是动态的
}

// PatchFlag类型:
// 1  = TEXT         动态文本
// 2  = CLASS        动态class
// 4  = STYLE        动态style
// 8  = PROPS        动态props
// -1 = HOISTED      静态提升，完全不参与Diff
// -2 = BAIL         整个退出优化模式
```

```plaintext
Vue2: 不管节点是静态还是动态，Diff时全部比较一遍
Vue3: 编译时标记好，Diff时只看带PatchFlag的动态节点，静态的直接跳过

举例:
<div>
  <p>我是静态文本，永远不变</p>       ← Vue3直接跳过
  <p>{{ message }}</p>              ← Vue3只Diff这个
  <p>我也是静态的</p>                ← Vue3直接跳过
</div>

Vue2: 比较3个节点
Vue3: 只比较1个节点  → 快了3倍
```

## 面试回答模板

> Vue2 用的是**双端比较**，设4个指针从新旧列表两端向中间收拢，每轮按头头、尾尾、头尾、尾头的顺序匹配，匹配不上就用 key 去旧列表里查。
>
> Vue3 做了优化：先从头和尾预处理掉相同的节点，剩下中间乱序部分，用**最长递增子序列(LIS)** 算出哪些节点不需要移动，从而把 DOM 移动次数降到最少。
>
> 另外 Vue3 在编译阶段还加了 **PatchFlag 静态标记**，Diff 时直接跳过静态节点，只处理动态节点，进一步提升了性能。
